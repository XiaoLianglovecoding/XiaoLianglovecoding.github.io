Go与Redis实战：实现高并发短信登录与自动注册

在传统的 Web 开发中，用户的登录态通常由服务器的 Session 来维护。但在分布式集群环境下，Session 共享会面临诸多痛点。所以，我使用 Go 语言（Gin 框架 + GORM）结合 Redis，完整实现了一套企业级的“短信验证码登录与自动注册”闭环方案。

下面复盘一下核心的架构思路与踩坑点。

1. 为什么验证码要存入 Redis？

HTTP 是无状态的，“发送验证码”和“点击登录”是两次完全独立的网络请求。为了跨请求核对验证码，我们必须找个“储物柜”。

弃用 MySQL 的原因是因为验证码是“阅后即焚”的垃圾数据，频繁的磁盘 I/O 极其拖慢性能；而 Redis 完美契合这一场景，尤其是它的 TTL（过期时间） 机制，天然支持“验证码 2 分钟有效”的业务需求。

// 存入 Redis 并设置 2 分钟 TTL，只需接收 error
err := s.redisClient.Set(ctx, key, code, 2*time.Minute).Err()
if err != nil {
    return result.Fail("验证码保存失败")
}

### 2. 核心业务：无感“自动注册”（Find or Create）
在登录验证通过后，系统需要校验该手机号是否已经是我们的用户。为了提升用户体验，我们采用“自动注册”逻辑：如果数据库中查不到该用户，说明是第一次来的新用户，系统会自动为其创建账号。

这里充分体现了 Go 语言严谨的错误处理哲学——通过精细的错误分类拦截，将“新用户注册”、“老用户登录”和“数据库异常”完美分流：

```go
// 1. 拿着手机号去数据库查人
user, err := s.userRepo.FindUserByPhone(ctx, phone)

// 2. 精准拦截：这个错误是不是因为“找不到记录”引起的？
if errors.Is(err, gorm.ErrRecordNotFound) {
    // 如果是，说明是新用户，在内存中组装新用户对象
    user = &model.User{
        Phone:    phone,
        // 利用 UUID 随机生成一个默认昵称，切片截取前 8 位以防过长
        NickName: "user_" + uuid.New().String()[:8], 
    }
    
    // 执行 INSERT 插入数据库
    // 此时 GORM 会自动填充嵌套的创建/更新时间戳，未赋值的字段会自动落入 MySQL 的“零值”
    createErr := s.userRepo.CreateUser(ctx, user)
    if createErr != nil {
        return result.Fail("创建新用户失败")
    }
    
} else if err != nil {
    // 3. 拦截真正的系统灾难（如数据库断网、宕机、硬盘满等）
    return result.Fail("查询用户异常")
}
// 4. 如果 err == nil，完美避开 if 拦截，代表是老用户，直接带着装满数据的 user 对象往下走




###3. 数据脱敏与存储：巧用 Hash 结构
登录成功后，我们需要将用户信息缓存到 Redis 中，以便后续请求可以实现快速鉴权。

如果直接将对象序列化为 JSON 字符串存入，未来如果用户仅仅想修改一个头像或昵称，都必须把整个 JSON 取出来、反序列化、修改、再重新序列化存入，在大并发下非常消耗 CPU。因此，我们选择 Redis 的 Hash（哈希表） 结构，将其看作一个键值对字典，支持对单个字段进行极轻量级的修改。

在组装 Map 时，包含了一个极其重要的大厂工程避坑点（防止大数精度丢失）：

```go
// 构建通用的万能复合容器，键为 string，值为任意类型 interface{}
userMap := map[string]interface{}{
    // 核心细节：数据库的 ID 通常是 int64（如分布式雪花ID）
    // JavaScript/前端在解析超过 15 位的超大数字时会出现精度丢失（最后几位变0）
    // 后端统一通过 strconv 强转为十进制字符串，是最安全、规范的工业做法
    "id":       strconv.FormatInt(user.ID, 10), 
    "nickName": user.NickName,
    "icon":     user.Icon,
}

// 写入 Redis Hash 结构
// go-redis 驱动非常智能，允许直接传入 map，它会在底层自动遍历平铺成 HSET 的标准命令格式
err = s.redisClient.HSet(ctx, tokenKey, userMap).Err()


###4. 鉴权屏障：RefreshToken 中间件
为了让用户的登录状态在后续请求中持续生效，同时避免用户频繁掉线，我们引入了 Gin 框架的中间件机制（其作用类似于 Java 中的拦截器拦截请求并挂载 ThreadLocal）。

该中间件的核心思路是：只要请求带了合法的 Token，就帮其刷新 Redis 的过期时间，并将用户信息注入请求上下文（Context），实现链路传递。

```go
package middleware

import (
    "time"
    "strconv"
    "[github.com/gin-gonic/gin](https://github.com/gin-gonic/gin)"
    "[github.com/redis/go-redis/v9](https://github.com/redis/go-redis/v9)"
    "go-hmdianping/internal/pkg/constants" // 替换为你真实的常量包路径
    "go-hmdianping/internal/dto"
)

// RefreshTokenMiddleware 是 Token 刷新中间件
// 通过闭包机制将服务启动时初始化的 redisClient 注入进来
func RefreshTokenMiddleware(redisClient *redis.Client) gin.HandlerFunc {
    return func(c *gin.Context) {
        // 1. 从 HTTP 请求头中尝试读取 Token
        token := c.GetHeader("authorization")
        if token == "" {
            // 如果没有 token，说明是未登录的游客，直接放行
            c.Next()
            return
        }

        // 2. 拼接 Redis Key
        key := constants.LoginUserKey + token

        // 3. 一把捞出 Redis Hash 中的所有键值对
        // 注意：go-redis 的 HGetAll 当 key 不存在时会返回空 map 而不报 redis.Nil 错，非常友好
        userMap, err := redisClient.HGetAll(c, key).Result()
        if err != nil || len(userMap) == 0 {
            // 查不到数据，说明 token 已过期或伪造，直接放行
            c.Next()
            return
        }

        // 4. 将查出来的字符串 Map 反向解析并还原为特定的数据传输对象 (UserDTO)
        id, _ := strconv.ParseInt(userMap["id"], 10, 64) // 将字符串 ID 还原回 int64
        userDTO := dto.UserDTO{
            ID:       id,
            NickName: userMap["nickName"],
            Icon:     userMap["icon"],
        }

        // 5. 【核心魔法】将对象塞进 Gin 的 Context 上下文中
        // 这样整个请求生命周期内的后续 Handler 和 Service 都可以直接无感读取
        c.Set("user", userDTO)

        // 6. 只要用户持续在操作系统，就为其 Token 续命（例如延长 30 分钟）
        ttl := time.Duration(constants.LoginUserTTLMinutes) * time.Minute
        redisClient.Expire(c, key, ttl)

        // 7. 放行，流转到下一站
        c.Next()
    }
}


当中间件把用户塞进上下文后，业务层（如 GET /user/me）只需要通过类型断言，就能极其优雅地把人提出来直接返回，整个过程完全脱离了昂贵的 MySQL 查表，性能表现极佳：
```go
// Me 返回当前登录用户
func (s *userService) Me(ctx context.Context) result.Result {
    // 1. 先将底层的 context 断言还原为 Gin 的原生上下文 *gin.Context
    ginCtx, ok := ctx.(*gin.Context)
    if !ok {
        return result.Fail("系统异常：无法获取请求上下文")
    }

    // 2. 尝试从上下文中取出 "user"
    userObj, exists := ginCtx.Get("user")
    if !exists {
        return result.Fail("用户未登录")
    }

    // 3. 再次进行类型断言，将空接口转换为具体实体的 DTO 类型
    userDTO, ok := userObj.(dto.UserDTO)
    if !ok {
        return result.Fail("用户信息解析异常")
    }

    // 4. 成功拿到，直接包装返回
    return result.OKWithData(userDTO)
}
